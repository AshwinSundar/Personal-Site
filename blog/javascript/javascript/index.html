June 2022

I think I'm getting closing to understanding what I don't like about Javascript. It has something to do with what I perceive to be an implicit disdain for technical rigor. I just read an entire GitHub issue thread about how tons of production sites broke in 2016 because a developer unpublished a widely-used dependency called left-pad. The reason he did it is fascinating, and worthy of its own article, so I won't get it into here.

Left-pad is shockingly simple - all it does is pad a string with a space on the left. It's possibly the second program many people ever wrote when they first learned how to program. Pretty much just an intellectual exercise, where the main focus is understanding the mechanics of the language.

Somehow, it became an NPM dependency that was used...a lot of times. But when it was unpublished, massive projects like Facebook's react-native and Babel stopped working.

Ok, so to summarize, it seems like the solution is to just prevent dependencies from being unpublished...right? Well sure, that would solve this type of weakness.

But what happens if you receive malicious code from a trusted party? Such as npm's issue 19883, where file permissions were modified in such a way that running sudo npm on Linux machines caused the entire OS to crash permanently, requiring a reinstallation of the operating system.

Who do you trust? Should you just write everything from scratch, all the way down to the kernel level? Probably not, unless you want to spend the rest of your career working on exactly one new problem, where you spend 99% of your time rebuilding your computer from the circuit board up. Incidents like this do provide some learning lessons though:

  1. Vet your dependencies You should vet every piece of code that you didn't write and ask yourself if you'd like to work with the person that wrote this code. Because that is actually what you're doing, even though you might never meet them, interact with them, or even know of each other. With npm, you now have thousands of colleagues you have to coordinate with, without ever speaking to them, getting to know them, or even knowing if they're still alive. And it's all built on blind-trust in all that is Open-Source. How insane is that? Yet for some reason, the Javascript community thinks this is the culmination of open-source - everything is so open that you never "re-invent" the wheel, instead opting to bolt to your Porsche any wheel with god knows how many sidewall defects in it and just speeding off.

  2. Vet your technologies You might have read the last paragraph and thought, "I have thousands of dependencies in my project. How can I vet all of them?". The answer is you can't. Even if you spent the hundreds of hours needed to evaluate every last one of those dependencies, you still don't know what the developer team behind that dependency wants to do (or doesn't want to do) next. They might be building a beautiful, error-free dependency and tomorrow decide that they want to usurp it for malicious purposes. Or outright delete it, as in the case of Azer and left-pad. I believe dependency-overload to be a common problem to a lot of programming languages, but blind-trust is something special to web development, for reasons still unknown to me. A language I'm diving into lately, Rust, seems to run a tighter ship. Rust dependencies have to be installed very intentionally, and if they go unused your compiler will throw a warning at you. And perhaps it's because of the nature of the projects I'm working on, but the dependencies I do end up using serve a very clear purpose, and solve a problem that would be much too time-consuming for me to solve on my own.

Javascript seems to lack the technical rigor that other languages inherently possess, and I wonder if it comes down to the fact that the language itself was invented haphazardly, and deceptively. The lore of Javascript is that it was invented in 10 days to provide a scripting language for the Netscape browser in the 90s. I'm sure the programmer behind it, Brendan Eich, was very technically talented at the time, but there is no way anyone can create a whole PROGRAMMING language in 10 days that reasonably covers all possible use cases for the next 25+ years and also behaves idiomatically.

The deception in the language goes right back to its name. Why was this new language named with a reference to the hottest language of the 90s, when it had absolutely nothing to do with it? Like the answer to all terrible decisions, it was money. Eich originally wanted to name the language LiveScript, Netscape decided that naming it JavaScript would help them sell their product more easily. This deception continues to manifest itself today - visiting any Javascript framework's website is like ...(analogy???)...the websites all seem so polished and fancy, it's very hard to get any real concrete or technical details about why their particular framework is so good, the whole site looks childish and gimmicky...keep thinking on this one

This one might be a stretch, but maybe programming communities exhibit their own version of [imprint theory](https://en.wikipedia.org/wiki/Imprinting_(organizational_theory), where the environment that the language was born in influences the community that uses it well beyond its early days. Javascript was born into haphazard programming and deception. Rust was born into a desire to be a better C++, so they always have a chip on their shoulder. I find the history of anything to be so fascinating for this reason, because it helps me understand any topic I'm learning about with far more clarity than I might if I just focus on the here and now.
